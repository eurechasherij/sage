## Problem Statement

AI coding agents (Claude Code and peers) systematically overengineer and ignore
better available options because they write code before researching. The defect
is behavioral, not informational: in every observed failure the agent had the
answer available and simply did not look.

Two concrete, real failures:
1. The project already had SWR installed. The agent ignored it and hand-rolled
   manual polling with a React `useEffect`.
2. Using Laravel Pennant, the agent invented a convoluted "beta user" workaround
   when Pennant ships a documented interception API for exactly that
   (https://laravel.com/docs/12.x/pennant#intercepting-feature-checks).

The failures are discovered only after the session finishes, costing ~1 hour of
agent refactor time, re-doing tests, and wasted CI / GitHub Actions spend when a
PR has already run.

## Demand Evidence

- Frequency: "mostly every session." This is not an edge case; it is the default
  behavior of agentic coding sessions.
- Cost per instance: "I will only notice when the session is finished, which will
  take another hour for the agent to refactor, costing testing and if the AI runs
  a PR it will cost github actions."
- Stated need: "it doesnt instinctively open up docs, what I want is an mcp and a
  skill too, so it will always do research before working."

This is the product-defining fact: the tool must run on every ticket, not be a
helper the agent may remember to call.

## Status Quo

Today the user works around it two ways, both inadequate:
- Manually telling the agent which packages to use (does not scale, easy to forget).
- Adding a per-framework MCP (e.g. a Laravel MCP) for docs. This does not scale
  across the many stacks the user works in (Laravel/PHP, JS/TS, React) and, more
  importantly, does not change the agent's instinct to research first.

The real competitor is "the agent codes immediately," not another product.

## Target User & Narrowest Wedge

Target user: the user and their dev team at atas.ai, running AI coding agents
across multiple stacks. Internal tool first.

Narrowest wedge: a `/work [ticket link, breakdown]` command that forces the agent
to research before writing code:
1. Ground in the project: read installed dependencies + lockfiles, read
   version-pinned docs for what is already there. (Catches both SWR and Pennant.)
2. If nothing suitable is installed, search the world (npm/Packagist/etc.),
   compare candidates, check freshness/health.
3. A flag controls whether the AI auto-adopts a package or stops and asks the
   human to decide.

## Constraints

- Multi-stack from day one: at least JS/TS (npm) and PHP (Composer). React-aware.
- Must work in greenfield projects, where nothing is installed and the agent will
  never reach for a package on its own.
- Language-agnostic data layer. No per-framework MCP zoo.
- Internal tool; no external distribution or UI required for v1.

## Premises

1. The core defect is behavioral, not informational. The heart of the product is a
   forcing function (the `/work` skill) that makes research a required, ordered
   step. Data layers serve that step. AGREED.
2. "Know thy project" (read installed deps/lockfiles + version-pinned docs before
   writing) is the cheapest, highest-hit-rate capability and needs no external
   service. AGREED.
3. World-search is in v1, but built as a THIN AGGREGATOR over deps.dev + OSV +
   npm/Packagist/GitHub APIs (+ Context7 for docs). No proprietary index, no
   doc crawling at scale. This gives current, trustworthy data for safe
   "AI decides" mode while keeping v1 a ~1-week build. AGREED (refined from an
   original proposal to defer the service entirely).
4. The "AI decides vs human decides" flag belongs in v1 — a cheap gate in the
   skill flow. AGREED.
5. Architecture is one behavior layer (skill) + one language-agnostic data layer
   (MCP), not a per-framework MCP per stack. AGREED.

## Approaches Considered

Completeness scale below: 10 = enforces "always research" + full research data;
7 = full research data, soft/convention gate; 3 = partial data, no gate.

### Approach A: Light — skill + 3 tools (minimal viable)
`/work` skill + a small MCP (scan_project_dependencies, search_packages,
check_package_health), docs via Context7, decision written as a markdown note.
S effort (~3-4 days human / ~half-day CC). Risk: research stays optional, so it
may not actually fix the skipping. Completeness 6/10.

### Approach B: Full research protocol (ideal architecture) — CHOSEN
TS MCP server with 7 tools, SQLite cache, `/work` skill ordering the gate, both
decide modes, written decision artifact. Thin aggregator over deps.dev/OSV/
registry/Context7. M effort (~1 week human / ~1-2 days CC). Risk: gate is
convention (an MCP tool cannot force the agent to call it). Completeness 8/10.

### Approach C: Protocol + hard enforcement (creative/lateral)
Everything in B plus a Claude Code PreToolUse hook that blocks Edit/Write on
source files until a research-decision artifact exists for the task. The only
option that literally delivers "always research first." L effort (~1.5 weeks
human / ~2-3 days CC). Risk: blocking-hook UX; Claude-Code-specific. Completeness
10/10. Designated the immediate next increment after B.

## Recommended Approach

Approach B. It ships the real research capability in about a week and proves
research changes outcomes on the SWR/Pennant cases before adding a hard gate.
C's enforcement hook is the planned next increment: once the research flow is
trusted, the hook turns the nudge into a wall and delivers "always" literally.
Starting with C risks building a blocking gate before the thing behind it is
proven, which is the kind of friction that gets disabled.

### MCP tools (v1)
- `scan_project_dependencies` — read package.json + lockfiles (npm/pnpm/yarn/bun)
  and composer.json/composer.lock; return installed packages, versions, ecosystem,
  direct/transitive.
- `find_existing_capability` — input a desired capability (e.g. "polling data
  fetch", "feature flag override"); return likely reusable installed packages with
  confidence + doc links. v1 mechanism (decided, not open): the agent (model)
  classifies which installed packages plausibly cover the capability, given the
  scanned dependency list + each package's registry description/keywords. No
  per-package hardcoding, no embeddings in v1. A small curated alias map
  (e.g. "data fetching" -> swr, react-query, tanstack-query) seeds common cases;
  everything else falls to model classification over the dep list. The alias map
  ships with the MCP server as `aliases.json`, shape
  `{ "<capability>": ["<pkg>", ...] }`, keyed per ecosystem.
- `get_versioned_docs` — input package + version; use Context7 when available,
  fall back to homepage/README/GitHub releases/official docs URL; return source
  URLs + version confidence. Honest caveat: version-accurate docs effectively
  depend on Context7. Without it, the bounded fallback returns latest README /
  releases, which may not match the installed version — `version confidence` must
  reflect this, and PHP/Packagist coverage is thinner than npm, so this path fires
  more often there.
- `search_packages` — input ecosystem + capability; npm registry + GitHub for JS,
  Packagist + GitHub for PHP, deps.dev where supported; return candidates with
  freshness, downloads/stars, maintenance signals.
- `check_package_health` — OSV for vulns, deps.dev for advisories/scorecards/dep
  graph, registry/GitHub for last release, open issues, license, deprecation.
- `compare_packages` — ranked comparison with explicit tradeoffs and raw evidence
  (not a single opaque score). v1 decision: SHIP THIS MERGED into
  `search_packages`'s output (it returns ranked candidates with evidence), so v1
  is 6 tools, not 7. Split into a standalone tool only if comparison logic grows.
- `record_research_decision` — compact decision note: reused / added / avoided a
  dependency, reason, docs checked, risk.

### `/work` skill steps
1. Parse ticket + breakdown; extract required capabilities. The model produces a
   free-text list of capability strings (e.g. ["client-side data fetching with
   caching", "feature flag with per-user override"]); these strings are the input
   contract for `find_existing_capability` and `search_packages`.
2. `scan_project_dependencies`.
3. `find_existing_capability` per capability.
4. `get_versioned_docs` on installed hits.
5. Summarize reuse candidates BEFORE writing code.
6. If the ticket already names specific packages to use, skip the search step but
   still research them: `get_versioned_docs` + `check_package_health` on the named
   packages, surface findings, follow the human's choice. (Ticket content is
   user-trusted input: it can name packages and describe the task; it still goes
   through the health floor below before an auto-install.)
7. Else if nothing suitable is installed: `search_packages`.
8. `check_package_health` on top candidates.
9. `compare_packages` (merged into search output, see tools above).
10. Respect mode (see defaults below): `--human-decides` stops and asks with ranked
    options; `--ai-decides` chooses, subject to the auto-install safety floor.
11. `record_research_decision`, then implement.

### Flag defaults, decision artifact, and task identity
- Research is NOT skippable (decided). The `/work` gate always runs; the only thing
  the flag controls is who DECIDES, not whether research happens.
- Default mode (decided): decisions default to `--human-decides` — the agent stops
  and asks with ranked options. The agent only auto-decides when the user has
  explicitly opted in (a `--ai-decides` flag or a stored config). Reuse of an
  already-installed package is the one low-risk case where `--ai-decides` is the
  natural choice; adding a NEW dependency always requires the explicit opt-in.
- Auto-install safety floor (decided): even under `--ai-decides`, the agent must
  NOT auto-add a package that fails a hard floor — an OSV advisory at/above a
  severity threshold, deprecated, or below a maintenance floor (e.g. no release in
  N months / very low adoption). A candidate that fails the floor falls back to
  stop-and-ask regardless of `--ai-decides`. Ticket-named packages get the same
  floor check; a failure is surfaced to the human, not silently auto-installed.
- Skip signal (decided, observability): if implementation begins for a task with no
  `.sage/decisions/<task-key>.json` present, `/work` emits a warning/log line so a
  skipped gate is visible immediately rather than discovered an hour later. This is
  a soft signal in v1 (no blocking) and is the exact seam Approach C's hook hardens
  into a wall.
- Task identity (decided): a task is keyed by the ticket ID parsed from the
  `/work` argument; if absent, fall back to the current git branch name; if
  neither, a hash of the ticket/breakdown text. This key names the decision
  artifact and is what Approach C's hook will look for.
- Decision artifact (decided): `record_research_decision` writes
  `.sage/decisions/<task-key>.json` in the repo (plus a human-readable `.md`
  sibling). Fields: task key, per-capability outcome (reused | added | avoided),
  package + version, reason, docs URLs checked, health/risk summary, timestamp,
  mode used. This file is the audit trail and the artifact C's hook gates on.

### Reliability & degradation (decided)
Six of seven tools depend on external APIs, and this gate runs every session, so
failure handling is first-class, not an afterthought:
- GitHub API: read a token from `GITHUB_TOKEN`/`GH_TOKEN` env; if absent, warn
  once and degrade to registry-only signals (skip stars/issues), since
  unauthenticated GitHub is 60 req/hr.
- Any data API down / rate-limited / empty: the tool returns a typed
  `degraded` result naming which source failed; `/work` proceeds but records the
  gap in the decision artifact (e.g. `health: partial (OSV unavailable)`).
- Docs fallback chain is BOUNDED, not a crawler (honors premise #3): Context7 ->
  the single canonical docs URL or GitHub README for that package+version -> if
  both fail, record `docs_unavailable` and force `--human-decides` for that
  capability. No multi-page crawling.
- Cache (SQLite) serves stale-but-usable data when a live source is unavailable,
  with a per-source TTL; a stale hit is labeled as such in the artifact. SQLite
  runs in WAL mode; the cache is treated as single-writer per process and safe for
  concurrent readers, so sibling worktrees do not corrupt it.
- Per-source request timeout (default ~8s): a slow/hanging source converts to the
  same typed `degraded` result ("source unavailable: timeout") rather than taxing
  every ticket. The hot path never blocks on one slow API.

### Lean on vs build
- Lean on: Context7 (docs), deps.dev (cross-ecosystem metadata/advisories/scorecards),
  OSV (vulns), npm/Packagist/GitHub APIs (freshness/popularity), lockfiles as the
  primary source of truth.
- Build: the workflow gate, the dependency scanner, comparison normalization, the
  decision artifact, the cache.

## Open Questions

  suggesting. Open part: who owns and grows that curated map, and how often it is
  reviewed.
- `compare_packages` may be premature as a separate tool for v1 — its output
  overlaps `search_packages` + `check_package_health`. Open: ship it merged into
  `search_packages` output for v1 and split later, or build it standalone now.
- Per-source cache TTLs: concrete numbers (e.g. health 24h, docs 7d, search 6h)

- Which agent(s) beyond Claude Code must this support? Affects how soon C's hook
  (Claude-Code-specific) is worth building.

## Success Criteria

Note: Approach B's gate is convention (an MCP tool cannot force the agent to call
it), so v1 criteria are scoped to "when `/work` is invoked." Making the gate
unconditional is Approach C.

- When `/work` is invoked on the SWR example, it surfaces the installed SWR and
  its docs before any code is written, and the agent reuses it.
- When `/work` is invoked on the Pennant example, it surfaces the documented
  interception API before the agent writes a workaround.
- Measurable: across a replay of ~20 real recent tasks run through `/work`, the
  rate of "ignored an installed/documented better option" drops to near zero.
- A `record_research_decision` artifact (`.sage/decisions/<task-key>.json`) exists
  for every `/work` run, including runs that ended in degraded/partial data.

## Distribution Plan

SAGE is a HOSTED service + remote MCP, not a local install (decided).
- The service (search / health / docs aggregation + server-side SQLite/edge cache)
  is deployed on a free host (Cloudflare Pages, Supabase, or Vercel) under
  `sage.rematcha.dev`.
- It exposes a remote MCP endpoint at the same domain. Devs do a single MCP install
  (gstack-style) pointed at `sage.rematcha.dev` — no per-dev local Node server.
- The `/work` skill ships as a skill file in the team's skills directory.

Architecture consequence (decided): the hosted service cannot read local files, so
project-grounding splits. The agent/skill reads lockfiles locally (it has file
access) and passes the dependency list + capability strings to the remote tools;
the service does cross-referencing, world-search, health, and docs. `scan_project_
dependencies` therefore runs host-side (skill/agent), while `find_existing_
capability`, `search_packages`, `check_package_health`, `get_versioned_docs` run
server-side.

Because it is now a deployed web service, it has a real deploy pipeline (host's
git-push deploy). CI integration and PR-time enforcement (the original GitHub
Actions waste) remain a later increment, but are cheaper now: the same hosted
service can back a CI check (see Open Questions / roadmap).

## Eng Review (FULL) — Outcomes

Mode: FULL_REVIEW. 2 findings resolved + 2 cross-model tensions resolved. 0 critical gaps.

### Decisions applied
- §1 Architecture (P1): ALL model reasoning lives host-side (agent/skill does
  capability extraction, installed-package classification, docs-relevance, and the
  final decision). `sage.rematcha.dev` is a PURE STATELESS DATA API — no server-side
  LLM, no keys, no inference path. `find_existing_capability` therefore returns
  metadata/docs/health for the host model to rank, not a server-side verdict.
- §4 Performance (P2): the service exposes BATCH endpoints and the host
  parallelizes. Concrete: `resolveInstalledCapabilitiesBatch(deps[], caps[])`,
  `getDocsBatch(coords[])`, `checkHealthBatch(coords[])`. Cuts ~12 serial
  round-trips on a 4-capability ticket to ~2-3.
- 8A correction (honest split): v1's enforceable part is the stop-and-ask DEFAULT
  inside `/work` (when it runs, it asks and writes the artifact). v1 does NOT detect
  a *skipped* `/work` — nothing host-side can observe an edit without a PreToolUse
  hook, a wrapper, or a CI check. True skip-detection is the first enforcement
  increment (Approach C hook OR CI check), not v1. The doc no longer claims v1
  detects skips.
- Sequencing: the 20-task replay (the Assignment) runs FIRST, before building
  service depth. Build host-side scanner + skill + thin metadata service first;
  add world-search depth only to the degree the replay proves it is needed.

### Refinements folded from outside voice (eng-grade)
- Privacy classification is NOT by name shape: use manifest metadata, lockfile
  resolved URLs, workspace/path flags, and an allow/denylist. Scoped-npm/private-
  Packagist/GH-registry/mirrors can look public. Never probe a registry to test
  publicness (that leaks the name). Private names are used host-side for matching
  but never transmitted.
- Decision artifact stores: normalized lockfile hash (per manifest), normalized
  task hash, extracted-capabilities hash, tool/service version, policy version,
  degraded sources, timestamp, run id. A "valid" artifact must cover the current
  capability set, not just exist.
- Task identity = task key + content/run hash; the later hook/CI validates the
  latest *applicable* artifact, not mere file existence.
- Add an explicit "no dependency needed / simple code is fine" outcome and a
